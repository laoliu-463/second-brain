# 正确性校验规则

本文档定义 AI 整理结果的系统化正确性校验机制，与《整理检查清单》形成互补：
- **整理检查清单**：人工审核用的逐项检查表
- **正确性校验规则**：自动化/系统化的校验逻辑与技术实现

适用场景：
- AI 自动执行整理前的预校验
- 审核模式下的辅助判断
- 定期批量审计
- 系统化质量控制

---

## 1. 目录结构校验

### 1.1 校验目标

确保笔记被移动到正确目录，且移动操作安全可回滚。

### 1.2 校验规则

| 校验项 | 规则描述 | 风险等级 | 处理方式 |
|--------|----------|----------|----------|
| 目录存在性 | 目标目录必须在目录结构中存在 | 高 | 不存在则禁止移动 |
| 类型匹配 | 目标目录必须与笔记类型匹配 | 高 | 不匹配则禁止移动 |
| 目标空目录 | 移动到的目录应该是空目录或已有笔记 | 低 | 警告后允许移动 |
| 同名文件检查 | 目标目录不应存在同名文件 | 中 | 存在则要求重命名 |
| 层级深度 | 移动后路径深度不超过 5 层 | 低 | 超深则警告 |

### 1.3 类型-目录映射表

```
inbox → 01-收件箱/
daily → 02-日记/YYYY/YYYY-MM/
project → 03-项目/进行中/ | 03-项目/已暂停/ | 03-项目/已完成/
reading → 05-资源/阅读/
resource → 05-资源/资料/
knowledge → 06-永久笔记/
moc → 07-索引/
archive → 08-归档/
```

### 1.4 校验输出

```json
{
  "directory_check": {
    "passed": true/false,
    "target_directory": "03-项目/进行中/",
    "note_type": "project",
    "issues": [],
    "warnings": []
  }
}
```

## 2. 链接有效性检查

### 2.1 校验目标

确保所有内部链接指向真实存在的笔记，外部链接可访问。

### 2.2 内部链接校验

| 校验项 | 规则描述 | 风险等级 | 处理方式 |
|--------|----------|----------|----------|
| 目标存在性 | `[[笔记名]]` 指向的笔记必须存在 | 高 | 不存在则转为候选 |
| 链接格式 | 必须符合 `[[笔记名]]` 或 `[[笔记名|显示名]]` 格式 | 中 | 格式错误则修复或移除 |
| 歧义检测 | 多个相似笔记名存在时标记歧义 | 中 | 转为候选要求确认 |
| 自链接检测 | 笔记不应链接到自己 | 低 | 自动移除 |
| 孤儿链接 | 目标笔记已被删除时标记 | 中 | 转为候选或移除 |

### 2.3 外部链接校验

| 校验项 | 规则描述 | 风险等级 | 处理方式 |
|--------|----------|----------|----------|
| 链接格式 | 必须是有效的 URL 格式 | 高 | 格式错误禁止写入 |
| 可访问性 | 链接应能返回 200 状态码 | 中 | 不可访问标记"可能失效" |
| HTTPS 优先 | 应使用 HTTPS 而非 HTTP | 低 | 警告后允许 |
| 锚点检测 | 锚点链接应标记为可能不稳定 | 低 | 标记风险 |

### 2.4 校验输出

```json
{
  "link_check": {
    "internal_links": {
      "total": 5,
      "valid": 4,
      "candidates": 1,
      "broken": 0,
      "details": [
        {"link": "[[项目-第二大脑搭建]]", "status": "valid", "target_exists": true},
        {"link": "[[MOC-知识管理]]", "status": "candidate", "reason": "multiple matches"}
      ]
    },
    "external_links": {
      "total": 2,
      "valid": 2,
      "invalid": 0,
      "details": []
    }
  }
}
```

## 3. Frontmatter 完整性验证

### 3.1 校验目标

确保 frontmatter 必填字段完整，非必填字段不虚构。

### 3.2 字段必要性分级

| 字段 | 类型 | 必要性 | 说明 |
|------|------|--------|------|
| title | 通用 | 必填 | 必须存在 |
| type | 通用 | 必填 | 必须存在且为有效类型 |
| created | 通用 | 必填 | 首次创建时间 |
| updated | 通用 | 必填 | 最后更新时间 |
| date | daily | 必填（当 type=daily 时） | 日期 |
| status | project | 必填（当 type=project 时） | 项目状态 |
| source | reading | 必填（当 type=reading 且有来源时） | 来源链接 |
| topic | knowledge | 建议 | 主题标签 |
| source_note | 收件箱转正 | 必填 | 原始收件箱笔记 |

### 3.3 禁止虚构字段

以下字段在无依据时禁止自动填写：
- `author`（除非用户提供或来源明确）
- `aliases`（除非用户明确提供）
- `priority`（除非用户提供）
- `review_cycle`（除非用户提供）
- `area`（除非用户提供）
- `progress`（除非用户提供）
- `stakeholders`（除非用户提供）
- `deadline`（除非用户提供）

### 3.4 字段值有效性校验

| 字段 | 有效值 | 校验规则 |
|------|--------|----------|
| type | daily, project, reading, resource, knowledge, moc, inbox | 必须在允许值内 |
| status | active, paused, completed, archived, draft, candidate, permanent | 必须与类型匹配 |
| date | YYYY-MM-DD 格式 | 必须是有效日期 |
| confidence | high, medium, low | 仅用于 knowledge 类型 |
| ai_lock | true, false | 布尔值 |

### 3.5 校验输出

```json
{
  "frontmatter_check": {
    "passed": true,
    "required_fields_present": true,
    "forbidden_fields_filled": false,
    "issues": [],
    "warnings": ["field 'topic' recommended but not filled"]
  }
}
```

## 4. 内容质量校验

### 4.1 校验目标

确保正文内容质量符合标准，不擅自扩写，不伪造信息。

### 4.2 格式校验

| 校验项 | 规则描述 | 处理方式 |
|--------|----------|----------|
| 标题格式 | 标题应为 H1 级别 (# 标题) | 不符合则修复 |
| 段落分隔 | 段落间应有空行 | 缺少则警告 |
| 列表格式 | 列表应使用一致的标记 (- 或 1.) | 不一致则警告 |
| 引用格式 | 引用应使用 > 标记 | 不符合则警告 |
| 任务格式 | 待办应使用 - [ ] 格式 | 不符合则警告 |

### 4.3 内容一致性校验

| 校验项 | 规则描述 | 风险等级 | 处理方式 |
|--------|----------|----------|----------|
| 标题-正文一致性 | 标题应概括正文主题 | 中 | 不一致则警告 |
| 类型-内容一致性 | 类型判定应与正文语义匹配 | 高 | 明显冲突则降级 |
| 用户原意保留 | 不应改变用户原始表达 | 高 | 改变则标记问题 |
| 探索vs确定 | 探索性内容不应改为确定性结论 | 中 | 改变则警告 |

### 4.4 低幻觉校验

| 校验项 | 规则描述 | 风险等级 | 处理方式 |
|--------|----------|----------|----------|
| 事实添加检测 | 检测是否添加了原文没有的事实 | 高 | 发现则移除 |
| 来源归属检测 | 区分用户理解与外部摘录 | 中 | 未区分则警告 |
| 猜测标记 | 猜测性内容应标记为"可能" | 低 | 未标记则警告 |

### 4.5 校验输出

```json
{
  "content_check": {
    "format_passed": true,
    "consistency_passed": true,
    "hallucination_check_passed": true,
    "issues": [],
    "warnings": ["探索性表达被改为确定性结论"]
  }
}
```

## 5. 执行边界校验

### 5.1 校验目标

确保 AI 执行动作在允许的权限范围内。

### 5.2 权限级别定义

| 级别 | 可执行动作 | 典型场景 |
|------|------------|----------|
| L1 低风险 | 格式优化、标题优化、frontmatter 补全、候选链接建议 | 自动执行 |
| L2 中风险 | 目录移动建议、正文插链、创建新笔记 | 审核确认 |
| L3 高风险 | 删除笔记、合并笔记、拆分笔记、批量修改 | 明确授权 |

### 5.3 模式-权限映射

| 执行模式 | L1 | L2 | L3 |
|----------|----|----|-----|
| 保守模式 | 自动执行 | 需确认 | 禁止 |
| 审核模式 | 自动执行 | 需确认 | 需确认 |
| 自动模式 | 自动执行 | 自动执行 | 需确认 |

### 5.4 边界校验规则

| 校验项 | 规则描述 | 风险等级 | 处理方式 |
|--------|----------|----------|----------|
| 原始笔记保留 | 收件箱原始笔记必须保留 | 高 | 未保留则禁止执行 |
| 永久笔记保护 | 永久笔记正文不应被改写 | 高 | 改写则禁止 |
| 删除操作检测 | 检测是否有删除操作 | 高 | 有删除则转审核 |
| 合并操作检测 | 检测是否有合并操作 | 高 | 有合并则转审核 |
| 批量操作检测 | 检测是否涉及多个文件 | 中 | 批量则转审核 |

### 5.5 校验输出

```json
{
  "permission_check": {
    "passed": true,
    "current_mode": "review",
    "requested_actions": [
      {"action": "move_directory", "level": "L2", "requires_approval": true}
    ],
    "issues": [],
    "warnings": []
  }
}
```

## 6. 综合校验流程

### 6.1 校验顺序

```
1. 目录结构校验 → 2. Frontmatter 校验 → 3. 链接校验
       ↓                ↓                  ↓
    通过/失败        通过/失败            通过/失败
       ↓                ↓                  ↓
4. 内容质量校验 → 5. 执行边界校验 → 6. 综合判定
       ↓                ↓                  ↓
    通过/失败        通过/失败            合格/有风险/不合格
```

### 6.2 校验决策树

```
开始校验
    │
    ├─ 目录校验 ──失败──→ 标记问题，终止校验
    │     │
    │   通过
    │
    ├─ Frontmatter 校验 ──失败──→ 标记问题，继续但不执行
    │     │
    │   通过
    │
    ├─ 链接校验 ──失败──→ 标记问题，转候选
    │     │
    │   通过
    │
    ├─ 内容校验 ──失败──→ 标记问题，降级处理
    │     │
    │   通过
    │
    ├─ 权限校验 ──失败──→ 标记问题，阻止执行
    │     │
    │   通过
    │
    └─ 综合判定
          │
          ├─ 全部通过 ──→ 合格，可执行
          ├─ 有警告 ─────→ 有风险，审核后执行
          └─ 有问题 ─────→ 不合格，退回
```

### 6.3 校验输出模板

```json
{
  "overall_check": {
    "result": "pass/warning/fail",
    "score": 85,
    "can_execute": true,
    "requires_approval": false,
    "checks": {
      "directory": {"passed": true, "score": 100},
      "frontmatter": {"passed": true, "score": 100},
      "links": {"passed": true, "score": 90},
      "content": {"passed": true, "score": 95},
      "permission": {"passed": true, "score": 100}
    },
    "issues": [],
    "warnings": ["1 条候选链接未确认"],
    "recommendations": ["建议审核后执行"]
  }
}
```

## 7. 审计追溯机制

### 7.1 审计日志结构

每次整理操作应记录：

| 字段 | 说明 | 示例 |
|------|------|------|
| timestamp | 操作时间 | 2026-03-18T10:30:00Z |
| note_id | 笔记唯一标识 | inbox-20260318-001 |
| note_title | 笔记标题 | 项目进展记录 |
| operation_type | 操作类型 | organize/move/link/update |
| before_state | 操作前状态 | {directory, frontmatter, links} |
| after_state | 操作后状态 | {directory, frontmatter, links} |
| ai_model | 使用的 AI 模型 | claude-sonnet-4-6 |
| execution_mode | 执行模式 | review |
| risk_level | 风险等级 | medium |
| approved_by | 审批人（若需要） | user |
| issues_found | 发现的问题 | [] |

### 7.2 审计查询命令

```bash
# 查看今日所有整理操作
git log --since="2026-03-18" --until="2026-03-19" --oneline --all

# 查看特定笔记的整理历史
git log --all --full-history -- "01-收件箱/*.md"

# 查看 AI 整理相关提交
git log --grep="ai-" --oneline

# 查看特定目录的变更
git log -- 03-项目/进行中/ --oneline

# 查看可疑的大规模修改
git diff --stat HEAD~10..HEAD
```

### 7.3 回滚机制

| 回滚类型 | 适用场景 | 命令 |
|----------|----------|------|
| 单文件回滚 | 特定笔记整理出错 | `git checkout HEAD -- path/to/note.md` |
| 目录回滚 | 批量操作出错 | `git checkout HEAD -- 目录/` |
| 提交回滚 | 需要撤销整个提交 | `git revert <commit>` |
| 时间回滚 | 恢复到特定时间点 | `git checkout HEAD@{yesterday}` |

## 8. 校验触发机制

### 8.1 自动触发

| 触发条件 | 校验范围 |
|----------|----------|
| AI 整理前 | 目录、权限 |
| AI 整理后执行前 | 全部校验 |
| 定时任务（每日） | 链接、frontmatter |
| 手动触发 | 自定义范围 |

### 8.2 定期审计

建议每周进行一次全面审计：

```bash
# 审计脚本伪代码
每周审计任务:
  1. 检查所有内部链接有效性
  2. 检查所有外部链接可访问性
  3. 扫描 frontmatter 完整性
  4. 检查是否有孤儿笔记
  5. 统计本周整理操作
  6. 生成审计报告
```

## 9. 合格标准

### 9.1 量化指标

| 指标 | 合格标准 | 优秀标准 |
|------|----------|----------|
| 目录归位准确率 | >= 85% | >= 95% |
| 链接有效率 | >= 90% | >= 98% |
| Frontmatter 完整率 | >= 90% | >= 98% |
| 无伪造来源率 | 100% | 100% |
| 权限合规率 | 100% | 100% |

### 9.2 质量红线

以下任一情况出现，本次整理直接判定为**不合格**：
- 伪造参考资料链接
- 生成悬空内部链接
- 未经授权删除笔记
- 未经授权合并/拆分笔记
- 改变用户原始表达意图

## 10. 与《整理检查清单》的关系

| 维度 | 整理检查清单 | 正确性校验规则 |
|------|--------------|----------------|
| 定位 | 人工审核辅助 | 自动化/系统化校验 |
| 触发 | 审核时手动使用 | 可自动触发 |
| 粒度 | 逐项检查表 | 规则引擎 |
| 输出 | 定性判断 | 定量评分 + 问题列表 |
| 覆盖 | 全面（人工判断） | 可程序化部分 |

两者应配合使用：
1. AI 整理后先经过**正确性校验规则**自动检查
2. 发现问题则修正后重新校验
3. 通过自动校验后再由**整理检查清单**进行人工复核
4. 人工复核确认后方可执行

---

*本规则与《整理检查清单》共同构成完整的质量控制体系。*
