# 第二阶段：信息架构设计

## 1. 设计目标

在第一阶段需求分析基础上，本阶段把系统落成可执行蓝图，确保四件事可直接实施：

1. 目录稳定，不随笔记规模增长而失控  
2. 笔记类型清晰，新增内容有固定入口  
3. 元数据统一，可检索、可统计、可自动化  
4. 自动化边界明确，AI 只做辅助维护

## 2. 仓库目录结构（MVP）

```text
my second brain/
├─ 00-Inbox/                  # 快速捕捉，待处理
├─ 10-Daily/                  # Daily Notes
├─ 20-Projects/               # 项目笔记（进行中）
├─ 30-Areas/                  # 持续责任领域（工作/健康/学习等）
├─ 40-Resources/              # 阅读、课程、资料
├─ 50-Knowledge/              # 长期沉淀（常青笔记）
├─ 60-MOCs/                   # 内容地图与导航页
├─ 70-Reviews/                # 周/月/季回顾
├─ 80-Archive/                # 归档（完成项目、旧资料）
├─ 90-System/
│  ├─ templates/              # 模板
│  ├─ automation/             # 自动化脚本与配置
│  └─ logs/                   # 自动化运行日志（可选）
├─ .obsidian/
└─ README.md                  # 首页入口（或 60-MOCs/Home.md）
```

## 3. 笔记类型矩阵

| 类型 | 主要用途 | 推荐路径 | 命名规范 | 必填字段 |
|---|---|---|---|---|
| `daily` | 每日记录、待办、碎片输入 | `10-Daily/` | `YYYY-MM-DD.md` | `type,date,created,updated,tags` |
| `project` | 项目主页（目标/范围/状态） | `20-Projects/` | `P-<slug>.md` | `type,project,status,owner,created,updated,tags` |
| `reading` | 书籍/文章/课程笔记 | `40-Resources/` | `R-<slug>.md` | `type,source,author,created,updated,tags` |
| `idea` | 灵感卡片（待提炼） | `00-Inbox/` | `I-YYYYMMDD-HHMM-<slug>.md` | `type,created,tags` |
| `knowledge` | 长期知识沉淀（可复用） | `50-Knowledge/` | `K-<slug>.md` | `type,topic,created,updated,tags` |
| `moc` | 主题导航/索引页 | `60-MOCs/` | `MOC-<topic>.md` | `type,topic,updated,tags` |
| `review` | 周/月回顾、复盘 | `70-Reviews/` | `Review-YYYY-[Www\|MM].md` | `type,period,created,updated,tags` |

## 4. 命名规范

1. 文件名统一使用英文 slug + 前缀，避免空格和中文全角符号  
2. 每类笔记使用固定前缀：`P-`、`R-`、`K-`、`MOC-`、`I-`  
3. Daily 仅用日期：`2026-03-17.md`  
4. 标题可中文，文件名建议英文 slug；通过 `title` 字段承载展示名  
5. 禁止随意重命名；重命名由人工确认后执行（避免链接抖动）

## 5. Frontmatter 字段体系

### 5.1 通用字段（所有笔记）

```yaml
---
id: ""                    # 可选：UUID 或短 ID
title: ""                 # 展示标题
type: ""                  # daily/project/reading/idea/knowledge/moc/review
status: ""                # active/on-hold/done/archived（按类型选填）
created: 2026-03-17
updated: 2026-03-17
tags: []
aliases: []
source: ""                # 外部来源链接（阅读/资料类常用）
project: ""               # 关联项目 ID 或项目名
area: ""                  # 责任领域
review_at: ""             # 下次回顾日期（可选）
ai_lock: true             # true = AI 不得改正文，仅建议
---
```

### 5.2 类型补充字段

1. `project`：`goal`, `deadline`, `stakeholders`, `next_action`  
2. `reading`：`author`, `publisher`, `year`, `url`, `rating`  
3. `knowledge`：`topic`, `related`, `confidence`  
4. `review`：`period`, `highlights`, `issues`, `decisions`

## 6. 模板体系（90-System/templates）

MVP 需要 6 个模板：

1. `tpl-daily.md`
2. `tpl-project.md`
3. `tpl-reading.md`
4. `tpl-idea.md`
5. `tpl-knowledge.md`
6. `tpl-review-weekly.md`

模板原则：

1. 先保证输入速度，再追求结构完整  
2. 模板正文保持简短，字段完整性放到 frontmatter  
3. 项目/回顾模板必须包含“下一步动作（Next Action）”

## 7. 页面流转关系

核心流转链路：

```text
Capture(00-Inbox / 10-Daily)
  -> Triage(每日/每周整理)
  -> Distill(提炼为 project/reading/knowledge)
  -> Link(挂到 MOC 和相关笔记)
  -> Review(70-Reviews 周/月复盘)
  -> Archive(80-Archive)
```

导航结构建议：

1. `Home`（入口页）  
2. `MOC-Projects`（项目总览）  
3. `MOC-Areas`（责任领域总览）  
4. `MOC-Knowledge`（知识主题总览）  
5. `MOC-Reviews`（回顾入口）

## 8. GitHub 映射与版本策略

### 8.1 仓库策略

1. 单仓库管理整个 vault（MVP 阶段）  
2. GitHub 使用 private repo  
3. 提交粒度小而清晰，优先“单主题提交”

### 8.2 提交约定（建议）

1. `note:` 新增/更新普通笔记  
2. `daily:` 每日日志  
3. `moc:` 索引与导航更新  
4. `review:` 周/月复盘  
5. `chore(automation):` 自动化脚本或规则

### 8.3 .gitignore 基线（在现有基础上补充）

建议忽略：

1. `.obsidian/workspace*.json`
2. `.obsidian/cache/`
3. `.obsidian/workspace-mobile.json`
4. 系统临时文件（`.DS_Store`, `Thumbs.db`）

按需纳入版本控制：

1. `.obsidian/core-plugins.json`
2. `.obsidian/community-plugins.json`
3. 关键插件配置（例如 `obsidian-git/data.json`，若你希望多端一致）

## 9. 自动化节点与边界

允许 AI 自动化：

1. 生成索引页草稿（MOC、周回顾草稿）
2. 检查坏链、孤立笔记、缺失字段
3. 给出标签/归类建议
4. 批量补全 `updated`、`review_at` 等低风险字段

必须人工确认：

1. 删除文件
2. 移动/重命名笔记
3. 修改核心正文观点
4. 批量改写超过 10 篇笔记正文

默认保护规则：

1. `ai_lock: true` 的笔记仅允许建议，不允许直接改正文  
2. 自动化改动必须有 Git 提交记录  
3. 高风险动作先生成 diff 预览再执行

## 10. 本阶段验收标准

满足以下条件即完成第二阶段：

1. 目录树已创建并可用  
2. 6 类模板已落地  
3. 至少 1 个 Home + 3 个 MOC 页面可导航  
4. frontmatter 字段在新建笔记中稳定出现  
5. 自动化边界写入规则文件并可执行  
6. GitHub 私有仓库可正常 push/pull
